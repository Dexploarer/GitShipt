## What The System Does

GitBags is a Bags.fm/Solana launchpad for GitHub repositories. A repo owner signs in with GitHub, chooses a repository, configures token metadata and leaderboard rules, launches a Bags token, and lets GitBags index repo activity. Daily workflows rank contributors, freeze snapshots, claim accrued Bags fees to a platform pool wallet, and redistribute SOL to linked contributor wallets or escrow.

The user-facing product currently includes:

- Public landing page with featured GitBags project and platform KPI ticker.
- Public explore page with filters and project cards.
- Public global leaderboard.
- Public project pages under `/r/[org]/[repo]` with sidebar, token stats, leaderboard, pool overview, payout feed, snapshots, repository, token, and docs subroutes.
- Public contributor profiles under `/u/[username]`.
- Launch wizard under `/launch`.
- Auth pages for GitHub sign-in and wallet linking.
- Dashboard/account pages for projects, wallets, earnings, security, and API keys.
- Project owner console with settings, scoring, payouts, repository install/reindex, token, team, docs, and API keys.
- Super-admin console with ops, projects, users, payouts, snapshots, fees, treasury, workflows, integrations, audit, maintenance, abuse, feature flags, and DB inventory.
- API route handlers for projects, launch, GitHub App install, wallets/SIWS, claims, API keys, trading quote/swap, webhooks, admin actions, health, and cron triggers.
- Vercel Workflows for GitHub indexing, leaderboard compute, snapshots, payout execution, escrow expiry, claim processing, KPI publishing, and health pulse.

## How It Works

The app is a Bun workspace with `apps/web` as the Next.js 16.2.4 App Router app and three packages:

- `packages/ui`: shadcn-style primitives, sidebar primitives, and shared UI store.
- `packages/shared`: Zod schemas for API, GitHub, launch, payout, and API-key contracts.
- `packages/lib`: formatting, IDs, language colors, and utility helpers.

The data layer is Drizzle over Neon. `dbHttp` is used for serverless reads/writes and `dbPool` for multi-statement transactions. Redis backs rate limiting, SIWS nonces, cache snapshots, and idempotency. Auth is better-auth with GitHub OAuth and custom Solana wallet verification. Permissions are rechecked with `requirePermission`, and recent RLS work adds AsyncLocalStorage context and a full RLS migration.

Workflow flow:

1. `/api/cron/index-github` starts `indexGithubDeltas`.
2. `indexProjectDeltas` fetches commits and merged PRs through the GitHub App.
3. `computeLeaderboard` scores contributors and assigns ranks.
4. `/api/cron/snapshot` starts `takeSnapshot`.
5. `/api/cron/payout` starts `executePayout`.
6. `executePayout` checks kill switch and hot-wallet caps, claims Bags fees, creates payout recipients, dispatches SOL transfers or escrow, and writes audit logs.
7. `/api/claims/link` and `/api/claims/escrow` start `processClaim` to bind a contributor identity to a verified wallet and drain escrow.

## Detailed Findings

### Critical Issues

**C1. Dependency audit fails on a critical RCE class vulnerability and multiple high-risk transitive packages**  
Location: `apps/web/package.json:25-58`, `bun.lock:1127`, `bun.lock:3731`, `bun.lock:4255`, `bun.lock:1531-1533`, `bun.lock:1657`, `bun.lock:3107`  
Category: Security  
Impact: This is not launch-clean for a crypto app. `bun audit` reports 31 vulnerabilities, including critical `protobufjs` via Solana wallet adapter transitive dependencies. GitHub Advisory GHSA-xq3m-2v4x-88gg / CVE-2026-41242 describes arbitrary code execution when attackers can control protobuf definitions; patched versions are `7.5.5` and `8.0.1`. The lockfile has top-level `protobufjs@7.5.5`, but nested Trezor paths still resolve `protobufjs@7.4.0`.  
Recommendation: Add an override/resolution or upgrade/prune wallet adapters so no vulnerable nested protobuf remains; update `workflow`/Bags SDK paths for `undici`, `axios`, `serialize-javascript`, and `bigint-buffer` exposure. Re-run `bun audit` until clean or document accepted non-runtime dev-only risk.  
Suggested command: `/harden`, then `/security-best-practices`.

**C2. Snapshot creation is not idempotent by project/day, creating duplicate payout risk**  
Location: `apps/web/workflows/takeSnapshot.ts:35-40`, `apps/web/workflows/takeSnapshot.ts:59-66`  
Category: Financial correctness / Workflow idempotency  
Impact: The code states that the snapshot table lacks `(projectId, day)` uniqueness. `executePayout` dedupes payout rows per snapshot, not per project/day. If cron/manual retrigger creates two frozen snapshots for the same leaderboard period, both can be picked up as payout candidates. For a real pool wallet this can become duplicate distribution.  
Recommendation: Add a snapshot period key and unique index, make `freezeSnapshot` an upsert/no-op for existing periods, and key payout execution by `(projectId, snapshotPeriod)` in addition to `snapshotId`.  
Suggested command: `/harden`.

### High-Severity Issues

**H1. Light theme fails WCAG AA contrast on real rendered pages**  
Location: `apps/web/app/globals.css:92-120`, homepage `apps/web/app/(public)/page.tsx:67-76`, explore `apps/web/app/(public)/explore/page.tsx:60-67`, project stat labels through project components  
Category: Accessibility / Theming  
Impact: Axe reported serious color-contrast failures on `/`, `/explore`, and `/r/SYMBaiEX/gitbags`. Examples include `text-fg-muted` at contrast ~3.23 on `#fafafc`, small uppercase labels at ~3.37 on white, and success badge text at ~3.0 on `success-soft`. Users with low vision will struggle, and the design system claim that all foreground/surface combinations exceed 4.5:1 is not true in the rendered app.  
Recommendation: Recalibrate `--fg-muted`, semantic-on-soft tokens, and small-label usage in light mode. Keep disabled text separate from muted text.  
Suggested command: `/normalize`, then `/fixing-accessibility`.

**H2. Mobile project page footer occludes content**  
Location: `apps/web/app/r/[org]/[repo]/_components/ProjectShell.tsx:74-103`, `apps/web/components/public/PublicAppShell.tsx:41-75`  
Category: Responsive / UX  
Impact: In a 390x844 screenshot, the footer overlays the lower portion of the project page. The daily pool card's large `124.50 SOL` amount is visibly cut off. This makes the most important money figure unreadable on mobile.  
Recommendation: Add bottom safe-area padding to scroll containers, or make the footer part of normal flow on mobile. Verify `/`, `/explore`, `/r/*`, and long dashboard pages at 390px and 430px widths.  
Suggested command: `/adapt`.

**H3. CSP still allows production inline scripts and styles**  
Location: `apps/web/next.config.ts:21-41`  
Category: Security  
Impact: The app has strong baseline headers, but `script-src 'unsafe-inline'` weakens XSS containment. The comment correctly calls it a tradeoff, but a crypto app with wallet flows should not carry this indefinitely.  
Recommendation: Move to nonce-based CSP, keep inline style concessions as narrow as possible, and add CSP reporting before enforce.  
Suggested command: `/harden`.

**H4. SPL/token escrow drains are marked drained without on-chain transfer**  
Location: `apps/web/workflows/steps/escrow-helpers.ts:43-59`, `apps/web/workflows/steps/escrow-helpers.ts:164-180`  
Category: Financial correctness  
Impact: Non-native token escrow rows are marked `drained` with `spl-drain-not-implemented-v0`. If any future path writes token escrow, the UI/audit trail will say funds drained even though no transfer occurred.  
Recommendation: Refuse non-native token escrow drains with a failed/pending state until implemented; do not mark as drained.  
Suggested command: `/harden`.

**H5. Admin and ops surface breadth exceeds completed functionality**

## Patterns & Systemic Issues

- **Strong architecture, uneven completion**: the repo has the right feature map, but several surfaces are v1.1/read-only while still appearing in the full navigation.
- **Security architecture is thoughtful but dependency hygiene is currently failing**: in-route auth, RLS context, audit logs, cron auth, and CSP are good; dependency audit is not.
- **Design tokens are used consistently, but rendered accessibility is not yet guaranteed**: theme lint passes while axe fails contrast.
- **Stub/simulated mode is useful and honest in places, but needs one unified product language**: `simulated_live`, missing Bags key messaging, fake volume, and stub payout states should be impossible to mistake for production values.
- **Workflow idempotency is partly excellent and partly incomplete**: external payout steps use `getStepMetadata().stepId`, but snapshot period uniqueness is missing.

## Positive Findings

- The route tree is substantially built out and matches the PRD shape.
- Next.js 16 `proxy.ts` is correctly treated as redirects only, with comments warning that route-level auth is the security boundary.
